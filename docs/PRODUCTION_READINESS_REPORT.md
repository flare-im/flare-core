## 🔧 关键技术决策

### 1. 为何Mutex在低竞争下更快？

**原因分析**:
- Rust的 `parking_lot::Mutex` 在无竞争时为**快速路径**（fast path）
- 仅需1-2个CPU指令即可获得锁
- `AtomicU64` 的 `fetch_add` 虽然无锁，但仍有memory barrier开销
- 在低竞争场景，Mutex的优化效果显著

**最佳实践**:
```rust
// 低并发（< 1000连接）：使用 Mutex
stats: Arc<Mutex<ConnectionStats>>

// 高并发（> 10万连接）：使用 Atomic
stats: Arc<AtomicStats>

// 超高并发（> 100万连接）：Atomic + 分片
stats: Vec<Arc<AtomicStats>> // 每个分片独立统计
```

### 2. 为何选择令牌桶算法？

**对比其他算法**:

| 算法 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 令牌桶 | 允许突发流量，平滑限流 | 实现稍复杂 | **生产环境首选** ✅ |
| 漏桶 | 流量绝对平滑 | 无法处理突发 | 严格限速场景 |
| 固定窗口 | 实现简单 | 边界突发问题 | 非关键场景 |
| 滑动窗口 | 精确限流 | 内存开销大 | 精确计费场景 |

**选择理由**:
- IM场景需要处理突发消息（如群聊爆发）
- 令牌桶允许短时间超过速率，但长期平均符合限制
- 实现开销低，性能优秀（3200万+ ops/秒）

## 🎓 关键学习与洞察

### 1. 性能优化不是银弹

**发现**: Mutex在低竞争场景下比Atomic更快

**启示**:
- 盲目追求"无锁"不一定提升性能
- 需要根据实际负载选择数据结构
- 优化前必须先做性能分析（profiling）

### 2. 令牌桶的精妙之处

**设计精髓**:
```rust
// 关键：使用整数避免浮点运算
const PRECISION: u64 = 1000;
tokens: AtomicU64  // 实际值 = stored_value / PRECISION
```

**优势**:
- 避免浮点运算（整数CAS更快）
- 精度可控（PRECISION可调整）
- 无锁并发安全

### 3. 分层限流的必要性

**场景**:
- **连接级限流**: 防止单个恶意用户占用资源
- **全局限流**: 防止整体系统过载

**实现**:
```rust
// 先检查连接级
if !per_conn_limiter.try_acquire(1) {
    return Err("连接级限流");
}

// 再检查全局级
if !global_limiter.try_acquire(1) {
    // 注意：需要归还连接级令牌
    return Err("全局限流");
}
```

## 🚨 已知限制与风险

### 1. 内存占用

**风险**: 1000万连接需要 ~163GB 内存

**缓解措施**:
- 实施连接池复用
- 动态通道容量（根据活跃度调整）
- 冷连接降级存储

### 2. 文件描述符限制

**风险**: 操作系统默认限制（如Linux默认1024）

**解决方案**:
```bash
# 临时提升
ulimit -n 10485760

# 永久配置 /etc/security/limits.conf
* soft nofile 10485760
* hard nofile 10485760
```

### 3. 网络带宽

**计算**: 1000万连接 × 1KB/s = ~10GB/s

**建议**:
- 使用多网卡绑定
- 分布式部署
- CDN加速
